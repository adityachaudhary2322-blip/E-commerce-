PROJECT OVERVIEW
================================================================================
This is a full-stack E-Commerce Web Application with an optional Tkinter Desktop 
Admin Panel. The system allows users to browse products, add to cart/wishlist, 
place orders, track orders, cancel orders, and generate receipts. Admin can 
manage products (add, edit, delete, update stock/prices) through both web-based 
admin panel and Tkinter desktop application.


================================================================================
TECHNOLOGY STACK
================================================================================

1. BACKEND: Python 3.11
2. WEB FRAMEWORK: Flask (lightweight WSGI web framework)
3. FRONTEND: HTML5, Tailwind CSS, JavaScript
4. DESKTOP GUI: Tkinter (Python's standard GUI library)
5. DATA STORAGE: JSON files (no database required)
6. ICONS: Google Material Symbols

FLASK CONCEPTS USED:

1. ROUTING - Maps URLs to Python functions
   @app.route('/')                    # Home page
   @app.route('/api/products')        # API endpoint
   @app.route('/api/add-to-cart', methods=['POST'])

2. REQUEST PARSING - Gets data from frontend
   data = request.get_json()
   product_id = data.get('product_id')

3. RESPONSE FORMATTING - Returns JSON to frontend
   return jsonify({'success': True, 'products': products})

4. SESSION MANAGEMENT - Maintains user state
   session['user_id'] = user_id
   user_id = session.get('user_id')

5. CACHE CONTROL - Prevents caching of dynamic data
   @app.after_request
   def add_header(response):
       response.headers['Cache-Control'] = 'no-store'

================================================================================
OOP CONCEPTS IMPLEMENTED
================================================================================

1. CLASSES AND OBJECTS
   - Class: Blueprint/template for creating objects
   - Object: Instance of a class (actual entity in memory)

   EXAMPLE:
   class SimpleAdminPanel:        # Class definition
       def __init__(self, root):  # Constructor
           self.root = root       # Instance variable
   
   app = SimpleAdminPanel(root)   # Object creation

2. CONSTRUCTOR (__init__ METHOD)
   - Runs automatically when object is created
   - Initializes all instance variables
   - Sets up the UI and loads data

   EXAMPLE:
   def __init__(self, root):
       self.root = root
       self.products = []
       self.load_data()
       self.create_widgets()

3. ENCAPSULATION
   - Data hiding using methods
   - External code cannot directly access internal data

   EXAMPLE:
   def load_data(self):           # Public method
       self.products = load_products()  # Private data
   
   def save_data(self):           # Public method
       save_products(self.products)

4. INHERITANCE (in web version - product_management.py)
   - Child class inherits properties from parent class

   EXAMPLE:
   class Product(ABC):            # Parent (Abstract)
       pass
   
   class Electronics(Product):     # Child inherits from Product
       pass
   
   class Clothing(Product):        # Child inherits from Product
       pass

5. POLYMORPHISM (in web version)
   - Same method name, different behavior

   EXAMPLE:
   class Electronics(Product):
       def calculate_discount(self):
           return self.price * 0.10    # 10% discount
   
   class Clothing(Product):
       def calculate_discount(self):
           return self.price * 0.20    # 20% discount

6. ABSTRACTION (in web version)
   - Hiding implementation details
   - Abstract classes cannot be instantiated

   EXAMPLE:
   from abc import ABC, abstractmethod
   
   class Product(ABC):
       @abstractmethod
       def calculate_discount(self):
           pass

7. COMPOSITION
   - "Has-a" relationship between classes

   EXAMPLE (in web_app.py):
   app = Flask(__name__)                    # Flask app instance
   # The main application HAS-A data storage
   # The main application HAS-A cart manager
   # The main application HAS-A order manager

VIVA VOCE QUESTIONS & ANSWERS
================================================================================

Q1: Why did you choose Flask over Django?
A1: Flask is lightweight and perfect for small to medium projects. It gives more
    flexibility and doesn't enforce a specific project structure. Since our project
    doesn't need Django's built-in admin or ORM, Flask is the better choice.

Q2: How does the Tkinter admin panel sync with the website?
A2: Both applications read and write to the same JSON file (web_data/products.json).
    When admin makes changes, the file is updated. The website refreshes its data
    from the same file via API calls.

Q3: What is the purpose of session in Flask?
A3: Session maintains user state across HTTP requests. It stores the user_id after
    login, allowing the server to identify which cart and orders belong to which user.

Q4: Why use JSON files instead of a database?
A4: JSON is simpler to implement, requires no additional setup, and is sufficient
    for a project of this scale. It also makes the project portable - just copy
    the folder and everything works.

Q5: Explain the @app.after_request decorator usage.
A5: It adds cache-control headers to every response, preventing browsers from
    caching API responses. This ensures users always see the latest product data.

Q6: How does the quantity selector +/- work?
A6: incrementQty() and decrementQty() functions modify the input field value,
    checking against max stock. When user clicks "Add to Cart", it reads the
    current input value and sends that quantity to the server.

Q7: What happens when an order is cancelled?
A7: The order status changes to 'Cancelled', and the product stock is restored
    by adding back the quantities of cancelled items. The order remains in
    history but cannot be cancelled again.

Q8: How does COD payment status work?
A8: For COD orders, paymentStatus is set to 'Unpaid'. Only when the order
    status becomes 'Delivered' (after 2 days in demo), it changes to 'Paid'.

Q9: Why did you use both localStorage and Flask sessions?
A9: Flask session manages server-side cart/orders. localStorage manages
    client-side data like user authentication, addresses, and wishlist for
    demo purposes. This keeps the demo simple without a database.

Q10: What would you change for production deployment?
A10: Replace JSON with PostgreSQL, add proper user authentication with password
     hashing, implement HTTPS, add rate limiting, use environment variables for
     secrets, and deploy using Gunicorn + Nginx.


================================================================================
TROUBLESHOOTING GUIDE
================================================================================

Issue: Products not updating after admin panel change
Solution: Click refresh button on website or press Ctrl+F5

Issue: Receipt not showing
Solution: Check browser console for errors, ensure showReceipt() function exists

Issue: Cart not persisting after login
Solution: Cart is tied to session; logging in as guest then registering creates
         new session. Place order before logging out.

Issue: Tkinter admin panel won't open
Solution: Run 'python admin_panel.py' from terminal in the project directory

Issue: Flask server won't start
Solution: Port 5000 might be in use. Change port in web_app.py to 5001

Issue: Products have random IDs instead of sequential
Solution: Delete web_data/products.json and restart server to regenerate
